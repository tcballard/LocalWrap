# LocalWrap v3.0 PRD and Roadmap

## Working Title

**LocalWrap v3: the desktop cockpit for localhost apps**

## Product Promise

LocalWrap gets any local app to green, then brings the workspace back tomorrow.

LocalWrap should feel like the dependable desktop control room for local
development projects: import a repo, get to a working localhost URL, understand
what failed when it does not work, preview the app, and resume the same working
set later without terminal juggling.

## Status

- Stage: Draft implementation contract
- Target release: v3.0
- Owner: Tom Ballard
- Last updated: 2026-06-15

## Background

LocalWrap already has the core pieces of a localhost cockpit:

- Saved local projects with directory, command, port, URL, autostart, and
  open-on-ready preferences.
- Guided project import for package-script-based projects.
- Secure process start/stop/restart over Electron IPC.
- Readiness polling for local HTTP(S) URLs.
- Project Doctor checks for common launch problems.
- Bounded logs, copyable reports, tray controls, and in-app preview.
- A sample project for first-run onboarding.

The current product has evolved past "wrapper" language. v3.0 should make the
workflow obvious and habitual: first get a project to green, then restore that
workspace every day.

## Personas

### Primary: Solo or Full-Stack Developer

Juggles several local apps, ports, package managers, and terminal tabs. Wants a
fast way to start the right project, know when it is ready, and open or preview
it without remembering commands.

### Secondary: Maintainer, Demoer, Educator, or Support Engineer

Needs a repeatable way to run a local app and diagnose common setup failures
without deep terminal fluency.

### Not Primary for v3.0

Teams requiring shared cloud sync, production-like orchestration, container
management, access control, or central policy management.

## Goals

1. Make first successful launch feel guided and inevitable.
2. Promote Project Doctor from a status panel to an action-oriented path to
   green.
3. Make LocalWrap useful as a daily workspace restorer, not only a project setup
   utility.
4. Reposition the product around "desktop cockpit for localhost apps" in docs,
   release copy, and in-app language.
5. Harden the core enough that v3.0 is trustworthy as a relaunch.

## Non-Goals

- Do not become a Docker Desktop, OrbStack, VS Code Tasks, Raycast, or package
  manager replacement.
- Do not add cloud accounts, team sync, collaboration, or hosted services.
- Do not add a broad plugin architecture for v3.0.
- Do not weaken the Electron security model, command validation, local-only URL
  validation, CSP, context isolation, or IPC-only privileged actions.
- Do not block v3.0 on code signing, though release trust should remain a tracked
  risk.

## v3.0 Scope

### 1. First Green Run

The first-run and add-project experience should move from "configure, save, then
manually start" to a guided path with one obvious success outcome.

Required behavior:

- Provide a primary **Save & Start** path for draft projects.
- Preserve the existing safe save-before-start model.
- Show clear progress through:
  - Configured
  - Starting
  - Ready
  - Previewing
- Keep **Try Sample Project** as the lowest-friction demo path.
- Keep manual editing available for users who know their exact command, port, and
  URL.
- Avoid starting a draft project before validation succeeds.

Acceptance criteria:

- A new user can create the bundled sample project, start it, reach ready state,
  and preview it without needing to understand any hidden sequence.
- A valid imported project can be saved and started through one primary action.
- If validation blocks start, the UI explains the next step and leaves project
  state intact.
- Existing start/stop/restart behavior remains available for saved projects.

### 2. Action-Oriented Project Doctor

Project Doctor should become the v3.0 differentiator: the product does not only
run commands, it explains why a local app is not usable yet.

Required behavior:

- Make the top Doctor message a next-best-action summary.
- Preserve check-level detail for directory, command, dependencies, port, URL,
  process, and readiness.
- Improve dependency guidance with package-manager-specific install suggestions
  where LocalWrap can infer them safely.
- Keep safe actions explicit and user-initiated.
- Keep copyable Doctor reports for bug reports, support, and issue filing.

Candidate enhancements:

- Copy suggested install command.
- Copy suggested start command.
- Stronger port repair language when a port is busy.
- Better readiness timeout explanation: process running, URL not responding,
  wrong port, wrong host, or app still booting.

Acceptance criteria:

- Doctor output always answers: "What happened?" and "What should I try next?"
- Doctor actions never execute arbitrary project commands.
- Mutating fixes remain disabled while a project is running or has unsaved
  changes.
- Doctor reports include enough context for a maintainer to understand project
  config, runtime state, checks, recent timeline, and recent logs.

### 3. Resume Workspace

LocalWrap should become part of the daily local-dev ritual.

Required behavior:

- Track the last set of projects that were running together.
- Add a primary **Resume Workspace** action when no projects are running and a
  previous running set exists.
- Add or improve **Start All** and **Stop All** behavior from the dashboard and
  tray.
- Make tray copy describe running/ready/saved state clearly.
- Ensure resume skips projects that no longer validate, and routes those failures
  through Project Doctor.

Acceptance criteria:

- After stopping or quitting, LocalWrap can offer to restart the last running set.
- Resume handles partial failure without preventing other valid projects from
  starting.
- Users can see which projects resumed, which failed, and why.
- Stop-all remains safe and predictable across platforms.

### 4. Preview as a Working Surface

Preview should reinforce the cockpit metaphor without becoming a full browser.

Required behavior:

- Keep preview available only for ready local projects.
- Preserve sandboxed preview security and local-only navigation rules.
- Make preview state visible in the run progress path.
- Keep reload, open externally, and close controls.

Candidate enhancements:

- Responsive preview size presets.
- A more prominent default preview area after a project reaches ready state.
- Clearer preview failure state.

Acceptance criteria:

- Preview cannot access LocalWrap privileged APIs.
- Non-local navigation remains blocked or opened externally according to the
  current security model.
- Preview bounds remain stable during window resize and panel collapse/expand.

### 5. v3.0 Positioning and Release Assets

v3.0 should read as a product relaunch, not just a feature bump.

Required behavior:

- Update README language from "wrapper" toward "desktop cockpit for localhost
  apps."
- Align package description, GitHub description, README, and release notes around
  the same category language.
- Highlight:
  - First Green Run
  - Project Doctor
  - Resume Workspace
  - In-app Preview
  - Secure local control
- Produce a short demo flow for release assets:
  - Try Sample Project
  - Save & Start
  - Project Doctor
  - Ready
  - Preview
  - Resume Workspace

Acceptance criteria:

- A first-time reader can understand what LocalWrap is in one sentence.
- The README shows the v3 workflow before deep implementation details.
- Release notes explain why v3.0 exists.

### 6. Core Hardening

v3.0 should include enough engineering hardening to support the relaunch without
turning the release into an architecture rewrite.

Required behavior:

- Add atomic writes for project persistence.
- Preserve or back up corrupt project-store data instead of silently replacing it
  with an empty list.
- Make app shutdown semantics more explicit for running child processes.
- Add tests for new workspace resume behavior and persistence failure handling.
- Add at least one golden-path smoke test if feasible within the release window.

Candidate refactors:

- Split main-process modules by concern:
  - IPC handlers
  - tray
  - preview
  - updater
  - app lifecycle/bootstrap
- Extract renderer view-model helpers before adding substantial new UI state.

Acceptance criteria:

- Existing Jest suite passes.
- New behavior has focused tests.
- No security regression in preload, CSP, context isolation, command validation,
  or local URL validation.
- Shutdown does not knowingly orphan processes under normal quit paths.

## Out-of-Scope Backlog

These ideas are valid but should not block v3.0:

- Shared project config files committed to repos.
- Team workspaces and sync.
- Plugin marketplace or detector plugins.
- Full framework-specific setup automation.
- One-click dependency installation.
- Built-in terminal emulator.
- Docker/container lifecycle management.
- Public telemetry.
- Signed and notarized installers, unless available without derailing product
  scope.

## Implementation Principles

- Prefer narrow, testable modules in `lib/` for durable behavior.
- Keep privileged actions in the Electron main process behind explicit IPC.
- Keep renderer code free of filesystem, child-process, and shell access.
- Make unsafe or ambiguous actions explicit rather than automatic.
- Prefer copyable guidance over executing arbitrary commands.
- Keep v3.0 focused on workflow clarity and habit formation.

## Success Metrics

Because LocalWrap does not currently include telemetry, v3.0 success should be
judged through observable project and release signals:

- New users can complete the sample project flow without guidance.
- Maintainers can reproduce a golden path from release notes.
- Fewer issue reports need basic clarification about command, port, URL, or logs
  because Doctor reports include that context.
- README and release assets communicate the product in one sentence.
- v3.0 generates more external interest than recent v2 releases, measured by
  GitHub stars, release downloads, issues, discussions, and direct feedback.

## Open Decisions

Recommended defaults are included so implementation can proceed unless a
decision is explicitly changed.

| Decision                | Recommended default                                                         | Rationale                                                                                                     |
| ----------------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Primary draft action    | **Save & Start**                                                            | Clear, literal, and consistent with the safe save-before-start model.                                         |
| Product phrase          | Use **First Green Run** in planning, but avoid overusing it in the UI       | Good internal shorthand; UI should stay plain and action-oriented.                                            |
| Workspace model         | Global last-running workspace only                                          | Named groups are useful, but they add information architecture and are better after v3.0 validates the habit. |
| Resume source           | Last manually running set, updated when projects start/stop and before quit | Matches user intent better than only restoring from quit or stop-all.                                         |
| Preview presets         | Defer unless cheap after the main workflow lands                            | Useful polish, not central to v3.0's promise.                                                                 |
| Required refactor depth | Refactor only the modules touched by v3.0 behavior                          | Avoid turning the release into an architecture project.                                                       |
| Release signing         | Parallel admin track, not a v3.0 blocker                                    | Trust matters, but v3.0's product contract should not depend on signing logistics.                            |

## Initial Task Breakdown

1. Add persistence hardening.

3. Add start-all, stop-all, and resume-workspace lifecycle APIs.
4. Add **Save & Start** renderer flow.
5. Add progress state mapping for Configured, Starting, Ready, Previewing.
6. Update Project Doctor summaries and copyable report content.
7. Add tests for new lifecycle, persistence, Doctor, and renderer behavior.

## Definition of Done for v3.0

- The product visibly fulfills: "get any local app to green, then bring the
  workspace back tomorrow."
- First-run sample flow works.
- Imported project **Save & Start** flow works.
- Project Doctor provides next-step guidance on common failures.
- Resume Workspace works with partial failure handling.
- Tests pass.
- Security posture is preserved.
- README and release notes tell the v3.0 story clearly.
